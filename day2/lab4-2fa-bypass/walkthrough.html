2FA Bypass With an Unbound Code

This lab has a second login step for two-factor authentication, but the server checks the code and the username/password independently. That makes it possible to mix your code with someone else’s credentials.

Quick reminder

A 2FA code must belong to the same account that is trying to complete login. If the app validates a code without tying it to the user, the second factor stops being a second factor.

Where the vulnerability comes from

The app verifies the code and the credentials separately, then allows login if both checks pass.

code_valid = conn.execute(
    "SELECT * FROM tfa_codes WHERE code=? AND expired=0",
    (code,)
).fetchone()

user_valid = conn.execute(
    "SELECT * FROM users WHERE username=? AND password=?",
    (username, password)
).fetchone()

if code_valid and user_valid:
    # login succeeds
<input type="hidden" name="username" value="{{ username }}">
<input type="hidden" name="password" value="{{ password }}">

How the exploit works

Trigger a 2FA code for your own account, then intercept the second-step request and swap in the target user’s credentials while keeping your valid code.

Legitimate second-step request

POST /login2
username=user
password=password
2fa_code=483921

Tampered request

POST /login2
username=admin
password=adminpass
2fa_code=483921

The code still passes because the app only checks whether it exists. The credentials also pass, so the login completes as admin.

Fix

Bind the code check to the username so the code must belong to the same user whose credentials are being submitted.

code_valid = conn.execute(
    "SELECT * FROM tfa_codes WHERE code=? AND username=? AND expired=0",
    (code, username)
).fetchone()

Once the query includes username=?, a code issued for user can no longer be replayed with admin's credentials.
