SQL Injection Through the Login Form

This lab builds the login query by concatenating the username and password directly into SQL. The app also shows the final query when a login fails, which makes the injection path very easy to observe.

Quick SQLi Reminder

SQL injection happens when user input is merged into a SQL statement as code instead of being handled as data. If the database parses attacker input as part of the query, the attacker can change the logic.

In this lab the impact is authentication bypass. The goal is to turn the WHERE clause into something that always evaluates to true.

Where the vulnerability comes from

The login route builds the SQL query with an f-string. No placeholders are used, so the database receives raw user input inside the statement.

if request.method == 'POST':
    username = request.form['username']
    password = request.form['password']

    query = f"SELECT * FROM users WHERE username='{username}' AND password='{password}'"
    user = conn.execute(query).fetchone()

Because the app leaks the executed query on failed logins, you can see exactly how your payload changes the SQL statement.

Exploitation and fix

Use a valid username like admin and inject into the password field with ' OR 1=1 --.

Input

username=admin
password=' OR 1=1 --

Built query

SELECT * FROM users WHERE username='admin' AND password='' OR 1=1 --'

The injected OR 1=1 makes the condition true, and the trailing quote is neutralized by the SQL comment. The application then logs you in with the first row returned by the database.

Fix

The fix should focus on two things: validating the input before building the query, and using prepared statements so the database treats the values as data.

1. Input sanitization

Input validation does not replace prepared statements, but it helps reject clearly invalid values before they reach the database.

import re

def is_valid_username(value):
    return re.fullmatch(r'[A-Za-z0-9_]+', value) is not None

if not is_valid_username(username):
    return render_template('login.html', error='Invalid username format')

For a login form, this is useful for fields like usernames that should follow a known format.

2. Use prepared statements

This is the real fix. The query keeps placeholders, and the values are passed separately.

query = "SELECT * FROM users WHERE username=? AND password=?"
user = conn.execute(query, (username, password)).fetchone()

With this version, a payload like ' OR 1=1 -- is handled as a plain string value, not as SQL code.
