Race Condition in a Bank Transfer

MiniBank reads the sender's balance, does some extra work, then applies the update. Two requests sent at nearly the same time both read the same balance, both pass the check, and both subtract — leaving the sender with a negative balance.

Quick TOCTOU reminder

TOCTOU stands for time-of-check / time-of-use. A race condition shows up whenever a check and the action based on that check are separated in time and another thread can change state between them. On the web this usually means two HTTP requests racing to read-then-write the same row.

Where the vulnerability comes from

The transfer endpoint does the balance check and the debit as two separate operations, with extra queries in between.

# 1. READ the balance (time of check)
sender = conn.execute(
    "SELECT * FROM users WHERE username = ?", (user['username'],)
).fetchone()
recipient = conn.execute(
    "SELECT * FROM users WHERE account_id = ?", (target_account_id,)
).fetchone()

# 2. CHECK against the value we just read
if sender['balance'] < amount:
    return redirect(...)

# 3. extra work — widens the race window
conn.execute("INSERT INTO audit_log ...")

# 4. USE (time of use)
conn.execute("UPDATE users SET balance = balance - ? WHERE username = ?", (amount, sender['username']))
conn.execute("UPDATE users SET balance = balance + ? WHERE username = ?", (amount, recipient['username']))

The check in step 2 uses sender['balance'] — a Python variable read in step 1, not a live row lookup. SQLite is in WAL mode so concurrent connections can hold their own read snapshots at the same time. Two requests enter this code path, both read balance = 100, both pass the check, and both run the UPDATE balance = balance - 100. The update always decrements the current row, so the second one pushes the balance to -100.

Fun detail: changing UPDATE balance = balance - ? into UPDATE balance = ? WHERE ... with a pre-computed value makes this even worse — both threads would write the same 0 and the recipient's balance wouldn't even reflect both transfers correctly.

How the exploit works

Create two accounts (alice and bob) and log in as Alice. Copy Bob's 20-digit account id. Take your session cookie from the browser and fire a handful of transfers in parallel.

Parallel curls

for i in $(seq 1 5); do
  curl -s -X POST http://localhost:5006/transfer \
    -H "Cookie: session=<alice-session>" \
    -d "target_account_id=<bob-account-id>&amount=100" > /dev/null &
done
wait

Refresh the dashboard. You'll see multiple alice → bob transfers of $100 in the history, Alice's balance negative, and Bob's balance above $200. Alice only ever had $100.

Burp Intruder / Repeater

In Burp, capture the transfer request, send it to Repeater and create a tab group. Right-click and choose Send group in parallel (single-packet attack). That aligns the requests within a single TCP packet, which maximizes the chance they all pass the balance check before any of them writes.

Keep the burst around 5–10 requests. Flask's dev server is single-process, so huge bursts just turn into database is locked responses without helping.

Fix

The check and the write must be a single atomic statement, and the application must only commit the rest of the transfer if that statement actually updated a row.

# Debit only if the balance can cover the amount
cursor = conn.execute(
    "UPDATE users SET balance = balance - ? "
    "WHERE username = ? AND balance >= ?",
    (amount, sender_username, amount),
)
if cursor.rowcount == 0:
    return redirect(url_for('dashboard', message='Insufficient balance'))

conn.execute("UPDATE users SET balance = balance + ? WHERE username = ?", (amount, recipient_username))
conn.execute("INSERT INTO transfers ...")
conn.commit()

The WHERE balance >= ? moves the check into the database, so two concurrent UPDATEs cannot both succeed on the same starting balance. On a full-featured database you can also use SELECT ... FOR UPDATE inside an explicit transaction to lock the sender's row for the duration of the transfer.

Race conditions do not only apply to money. The same pattern breaks coupon redemption, invite-only signups, one-time 2FA bypasses, and "buy with points" flows. Anywhere you see "check X, then do Y", ask whether X could change between read and write.
